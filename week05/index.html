Notes

  • User errors are programmer errors tambien, because the program should be designed in a way that prevents the user from making the error.
  • An exception is an error that produces a return value that can then be used by the program to deal with the error.
  • A stack trace will work backwards from the point at which the error occurred to identify the original function or method that started the sequence.
  • Failing silently makes bugs hard to detect and hard to track down. The correct thing is that the errors are not silent and can be easily perceived, but not damaging the user experience.
  • Itâ€™s much better to spot errors early on, rather than have them cause problems later
  • It is recommended to use "use strict;" since we will have the advantage of knowing all the errors or possible errors while programming.
  • Placing 'use strict' at the beginning of a file will enforce strict mode on all the JavaScript in the file, and may cause errors if we are using third-party code. That is why it is more recommended to place all your code into a self-invoking function.
  • The recommended way to determine browser support for a feature is to use feature detection. This is done using an if statement to check whether an object or method exists before trying to actually call the method.
  • Debugging is the process of finding out where bugs occur in the code and then dealing with them.
  • An error object can be created to detect errors when we are using objects. It is quite a useful tool, especially if we know all the methods that are applied for specific errors.
  • The "Throw" statement will allow for any problems in your code to be highlighted and dealt with, rather than lurk quietly in the background. It will stop the execution of the program.
  • We can handle errors when we already have a production program, using "try, catch, and finally". This will cause the errors to still be displayed for the developers, but hidden from the user.
  • Test-driven development is considered to be best practice.
